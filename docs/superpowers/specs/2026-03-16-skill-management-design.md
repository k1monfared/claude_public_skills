# Skill Management System — Design Spec

## Overview

A shell-based CLI tool (`skill.sh`) for developing, managing, and distributing Claude Code skills from a central repository. Skills are developed in this repo and can be installed (copied) or linked (symlinked) into any project or the global `~/.claude/skills/` directory.

**Target users**: The repo maintainer and external users who clone the repo.

## Repository Structure

```
claude_public_skills/
├── skills/                    # All skills live here (source of truth)
│   ├── changelog/
│   │   └── SKILL.md
│   ├── explain/
│   │   └── SKILL.md
│   ├── loglog/
│   │   └── SKILL.md
│   └── review/
│       └── SKILL.md
├── templates/
│   └── SKILL.md               # Template for scaffolding new skills
├── manifest.json              # Auto-generated from SKILL.md frontmatter
├── groups.json                # Hand-maintained group definitions
├── skill.sh                   # The CLI tool
├── readme.log
└── CLAUDE.md
```

Key decisions:
- Skills live in a top-level `skills/` directory, not `.claude/skills/`. This is a source repo, not a consumer.
- `manifest.json` is always generated (never hand-edited). `groups.json` is hand-maintained.
- Skill templates live in `templates/` to keep them separate from actual skills.

### Migration from `.claude/skills/`

The existing skills in `.claude/skills/` will be moved to the top-level `skills/` directory as the first implementation step. The `.claude/skills/` directory will be removed from the repo. If the repo maintainer wants to use these skills locally during development, they can run `./skill.sh link --all --global` to symlink them into `~/.claude/skills/`.

## SKILL.md Frontmatter

Each skill's `SKILL.md` includes YAML frontmatter with metadata:

```markdown
---
name: changelog
description: Generate changelog entries from git commits
allowed-tools: Read, Write, Edit, Bash
argument-hint: [version or commit range]
tags: [git, documentation]
version: 1.0.0
---

# Changelog Skill

Use when asked to create or update a changelog...
```

Fields:
- **name** (required): Skill identifier, must match directory name.
- **description** (required): One-line summary, shown in `list` output.
- **allowed-tools** (optional): Claude Code field — preserved as-is during install/link. Not used by skill.sh.
- **argument-hint** (optional): Claude Code field — preserved as-is during install/link. Not used by skill.sh.
- **tags** (optional): For discoverability. Not used for grouping.
- **version** (required): Semver string. Used for tracking installed versions.

The existing skills will need `version` and optionally `tags` added to their frontmatter. The `allowed-tools` and `argument-hint` fields are Claude Code native fields — `build-manifest` extracts them into the manifest but skill.sh does not act on them. Any unrecognized frontmatter fields are preserved as passthrough.

## groups.json

Maps group names to skill lists. Hand-maintained by the repo maintainer.

```json
{
  "dev-tools": {
    "description": "Development workflow skills",
    "skills": ["changelog", "review"]
  },
  "documentation": {
    "description": "Documentation and explanation skills",
    "skills": ["explain", "loglog", "changelog"]
  }
}
```

- A skill can appear in multiple groups.
- The `validate` command checks that all referenced skills exist (error, not warning — see Validate).

## manifest.json

Auto-generated by `build-manifest`. Contains extracted metadata from all `skills/*/SKILL.md` frontmatter:

```json
{
  "skills": {
    "changelog": {
      "name": "changelog",
      "description": "Generate changelog entries from git commits",
      "allowed-tools": "Read, Write, Edit, Bash",
      "argument-hint": "[version or commit range]",
      "tags": ["git", "documentation"],
      "version": "1.0.0"
    }
  },
  "generated": "2026-03-16T12:00:00Z"
}
```

This file is a derived cache — the SKILL.md files are the source of truth. All frontmatter fields are included.

## CLI Commands

```
./skill.sh list                              # List all skills and groups
./skill.sh info <skill>                      # Show skill details
./skill.sh install <skill|group> <target>    # Copy skill(s) to target
./skill.sh link <skill|group> <target>       # Symlink skill(s) to target
./skill.sh uninstall <skill|group> <target>  # Remove skill(s) from target
./skill.sh new <skill-name>                  # Scaffold new skill from template
./skill.sh build-manifest                    # Regenerate manifest.json
./skill.sh validate                          # Check manifest and groups are in sync
./skill.sh init                              # Install git pre-commit hook
```

### Global Flags

- `--force` or `-f`: Skip confirmation prompts (overwrite without asking). Required for non-interactive/scripted use.

### Target Resolution

- `--global` or `-g`: expands to `~/.claude/skills/`
- `--all`: applies to all skills (used with `link` or `install`)
- A path (`.`, `/path/to/project`): expands to `<path>/.claude/skills/`

### Command Behaviors

**`list`**: Shows all skills with description and version. Shows all groups with their member skills. Runs lazy validation first.

**`info <skill>`**: Shows full metadata (description, version, tags, group membership).

**`install <skill|group> <target>`**: Copies skill directory to target. If a group name is given, installs all skills in that group. Conflict handling:
- Target does not exist: copy, create `.skill-source` marker.
- Target exists as a directory (not symlink): warn and prompt to overwrite (or overwrite with `--force`).
- Target exists as a symlink: warn that a link exists, prompt to replace with a copy (or replace with `--force`).

**`link <skill|group> <target>`**: Creates symlink at target pointing to skill in this repo. Conflict handling:
- Target does not exist: create symlink.
- Target exists as a symlink to the same source: no-op, already linked.
- Target exists as a symlink to a different source: warn, prompt to update (or update with `--force`).
- Target exists as a directory (not symlink): warn, prompt to replace with a link (or replace with `--force`). Refuses without `--force`.

**`uninstall <skill|group> <target>`**: Removes the skill directory/symlink from the target. If a group name is given, uninstalls all skills in that group. Checks for `.skill-source` marker or symlink before removing — refuses to delete skills it didn't install (override with `--force`).

**`new <skill-name>`**: Creates `skills/<name>/SKILL.md` from `templates/SKILL.md` with the name pre-filled. Errors if the directory already exists. Runs `build-manifest` automatically afterward.

**`build-manifest`**: Scans all `skills/*/SKILL.md`, parses YAML frontmatter, writes `manifest.json`.

**`validate`**: Checks:
- Skill directories on disk not in manifest → error
- Manifest entries with missing directories → error
- Groups referencing nonexistent skills → error
- Returns non-zero exit code on any error.

### YAML Frontmatter Parsing

The script uses a minimal line-by-line parser that handles the flat key-value patterns used in skill frontmatter:
- `key: value` (strings)
- `key: [a, b, c]` (inline arrays)

Nested YAML, multi-line values, and block arrays are not supported. This keeps the script dependency-free (no `yq` or `python` required). If a SKILL.md uses unsupported YAML syntax, `build-manifest` will warn and skip that field.

## Sync Safety — Three Layers

### 1. Manifest Generation

`build-manifest` scans `skills/*/SKILL.md`, extracts YAML frontmatter, writes `manifest.json`. Run automatically by the `new` command. Can be run manually anytime.

### 2. Lazy Validation on Every Command

Before `install`, `link`, `list`, `info` — a quick check runs:
- Any skill directories on disk not in manifest? Warn: "run `build-manifest`"
- Any manifest entries pointing to missing directories? Error and refuse to proceed.
- Any groups referencing nonexistent skills? Error and refuse to proceed.

This is lightweight — just stat checks and JSON parsing.

### 3. Pre-commit Git Hook

A standard git hook (`.git/hooks/pre-commit`), not a Claude Code hook:
- Runs `./skill.sh build-manifest` and stages `manifest.json`
- Runs `./skill.sh validate` — if validation fails, commit is blocked
- Guarantees the manifest is always correct in version control.

The `skill.sh` script provides a setup helper: `./skill.sh init` installs the git pre-commit hook. This is documented in the README and run once after cloning.

## Source Tracking

When `install` copies a skill, it writes a `.skill-source` file inside the installed skill directory:

```json
{
  "source": "claude_public_skills",
  "skill": "changelog",
  "version": "1.0.0",
  "installed": "2026-03-16",
  "method": "copy"
}
```

Purpose:
- `uninstall` verifies it's removing something it installed (won't delete manually created skills).
- Future `upgrade` command can compare installed version vs source version.
- Users can see where a skill came from.

For `link` mode, the symlink itself is the tracking — no marker file needed.

## Skill Template

`templates/SKILL.md`:

```markdown
---
name: SKILL_NAME
description: Brief description of what this skill does
allowed-tools: Read, Write, Edit, Bash
argument-hint: [optional arguments]
tags: []
version: 0.1.0
---

# SKILL_NAME

Description of when and how to use this skill.
```

The `new` command replaces `SKILL_NAME` with the provided name.

## Out of Scope (for now)

- `upgrade` command (compare installed vs source versions) — future enhancement
- Remote installation (install from URL without cloning) — future enhancement
- Skill dependencies (skill A requires skill B) — not needed yet
- Automated testing framework for skills — future enhancement
- `list` filtering by tag or group (e.g., `list --tag git`) — future enhancement
