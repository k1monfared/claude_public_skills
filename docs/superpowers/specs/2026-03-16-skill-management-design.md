# Skill Management System — Design Spec

## Overview

A shell-based CLI tool (`skill.sh`) for developing, managing, and distributing Claude Code skills from a central repository. Skills are developed in this repo and can be installed (copied) or linked (symlinked) into any project or the global `~/.claude/skills/` directory.

**Target users**: The repo maintainer and external users who clone the repo.

## Repository Structure

```
claude_public_skills/
├── skills/                    # All skills live here (source of truth)
│   ├── changelog/
│   │   └── SKILL.md
│   ├── explain/
│   │   └── SKILL.md
│   ├── loglog/
│   │   └── SKILL.md
│   └── review/
│       └── SKILL.md
├── templates/
│   └── SKILL.md               # Template for scaffolding new skills
├── manifest.json              # Auto-generated from SKILL.md frontmatter
├── groups.json                # Hand-maintained group definitions
├── skill.sh                   # The CLI tool
├── readme.log
├── CLAUDE.md
└── .claude/
    └── hooks.json             # Includes pre-commit manifest regeneration
```

Key decisions:
- Skills live in a top-level `skills/` directory, not `.claude/skills/`. This is a source repo, not a consumer.
- `manifest.json` is always generated (never hand-edited). `groups.json` is hand-maintained.
- Skill templates live in `templates/` to keep them separate from actual skills.

## SKILL.md Frontmatter

Each skill's `SKILL.md` includes YAML frontmatter with metadata:

```markdown
---
name: changelog
description: Generate changelog entries from git commits
tags: [git, documentation]
version: 1.0.0
---

# Changelog Skill

Use when asked to create or update a changelog...
```

Fields:
- **name** (required): Skill identifier, must match directory name.
- **description** (required): One-line summary, shown in `list` output.
- **tags** (optional): For discoverability. Not used for grouping.
- **version** (required): Semver string. Used for tracking installed versions.

## groups.json

Maps group names to skill lists. Hand-maintained by the repo maintainer.

```json
{
  "dev-tools": {
    "description": "Development workflow skills",
    "skills": ["changelog", "review"]
  },
  "documentation": {
    "description": "Documentation and explanation skills",
    "skills": ["explain", "loglog", "changelog"]
  }
}
```

- A skill can appear in multiple groups.
- The `validate` command checks that all referenced skills exist.

## manifest.json

Auto-generated by `build-manifest`. Contains extracted metadata from all `skills/*/SKILL.md` frontmatter:

```json
{
  "skills": {
    "changelog": {
      "name": "changelog",
      "description": "Generate changelog entries from git commits",
      "tags": ["git", "documentation"],
      "version": "1.0.0"
    }
  },
  "generated": "2026-03-16T12:00:00Z"
}
```

This file is a derived cache — the SKILL.md files are the source of truth.

## CLI Commands

```
./skill.sh list                              # List all skills and groups
./skill.sh info <skill>                      # Show skill details
./skill.sh install <skill|group> <target>    # Copy skill(s) to target
./skill.sh link <skill|group> <target>       # Symlink skill(s) to target
./skill.sh uninstall <skill> <target>        # Remove installed skill from target
./skill.sh new <skill-name>                  # Scaffold new skill from template
./skill.sh build-manifest                    # Regenerate manifest.json
./skill.sh validate                          # Check manifest and groups are in sync
```

### Target Resolution

- `--global` or `-g`: expands to `~/.claude/skills/`
- A path (`.`, `/path/to/project`): expands to `<path>/.claude/skills/`

### Command Behaviors

**`list`**: Shows all skills with description and version. Shows all groups with their member skills. Runs lazy validation first.

**`info <skill>`**: Shows full metadata (description, version, tags, group membership).

**`install <skill|group> <target>`**: Copies skill directory to target. If a group name is given, installs all skills in that group. If skill already exists at target, warns and asks to overwrite. Creates a `.skill-source` marker file in the installed skill directory.

**`link <skill|group> <target>`**: Creates symlink at target pointing to skill in this repo. If a non-symlink copy already exists at target, warns. No marker file needed — the symlink itself is the tracking.

**`uninstall <skill> <target>`**: Removes the skill directory/symlink from the target. Checks for `.skill-source` marker or symlink before removing — refuses to delete skills it didn't install.

**`new <skill-name>`**: Creates `skills/<name>/SKILL.md` from `templates/SKILL.md` with the name pre-filled. Runs `build-manifest` automatically afterward.

**`build-manifest`**: Scans all `skills/*/SKILL.md`, parses YAML frontmatter, writes `manifest.json`.

**`validate`**: Checks:
- Skill directories on disk not in manifest → warn
- Manifest entries with missing directories → error
- Groups referencing nonexistent skills → warn
- Returns non-zero exit code on errors.

## Sync Safety — Three Layers

### 1. Manifest Generation

`build-manifest` scans `skills/*/SKILL.md`, extracts YAML frontmatter, writes `manifest.json`. Run automatically by the `new` command. Can be run manually anytime.

### 2. Lazy Validation on Every Command

Before `install`, `link`, `list`, `info` — a quick check runs:
- Any skill directories on disk not in manifest? Warn: "run `build-manifest`"
- Any manifest entries pointing to missing directories? Error and refuse to proceed.
- Any groups referencing nonexistent skills? Warn.

This is lightweight — just stat checks and JSON parsing.

### 3. Pre-commit Git Hook

- Runs `./skill.sh build-manifest` and stages `manifest.json`
- Runs `./skill.sh validate` — if validation fails, commit is blocked
- Guarantees the manifest is always correct in version control.

## Source Tracking

When `install` copies a skill, it writes a `.skill-source` file inside the installed skill directory:

```json
{
  "source": "claude_public_skills",
  "skill": "changelog",
  "version": "1.0.0",
  "installed": "2026-03-16",
  "method": "copy"
}
```

Purpose:
- `uninstall` verifies it's removing something it installed (won't delete manually created skills).
- Future `upgrade` command can compare installed version vs source version.
- Users can see where a skill came from.

For `link` mode, the symlink itself is the tracking — no marker file needed.

## Skill Template

`templates/SKILL.md`:

```markdown
---
name: SKILL_NAME
description: Brief description of what this skill does
tags: []
version: 0.1.0
---

# SKILL_NAME

Description of when and how to use this skill.
```

The `new` command replaces `SKILL_NAME` with the provided name.

## Out of Scope (for now)

- `upgrade` command (compare installed vs source versions) — future enhancement
- Remote installation (install from URL without cloning) — future enhancement
- Skill dependencies (skill A requires skill B) — not needed yet
- Automated testing framework for skills — future enhancement
